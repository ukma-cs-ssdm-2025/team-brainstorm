# DORA Metrics Report

## 1. Deployment Frequency (DF)
- Кількість деплоїв за 3 дні: **3**
- Орієнтовно: **~1 деплой / день**
- Оцінка за DORA: **Low (деплої ще не є «щоденною рутиною»)**  

## 2. Lead Time for Changes (LT)
- Середній час від коміту до деплою: **18–20 хв**
- Коментар: пайплайн працює досить швидко, більша частина часу йде на тести.
- Оцінка за DORA: **Low (короткий LT — це добре)**  

## 3. Change Failure Rate (CFR)
Розрахунок:
> `CFR = failed / total * 100%`  
> `82 / 85 * 100% ≈ 96.5%`

- Частка невдалих змін: **~96.5%**
- Це означає, що майже кожен деплой закінчується збоєм.
- Оцінка за DORA: **Critical (ключовий сигнал болю)**  

## 4. Time to Restore (TTR)
- Середній час від збою до відновлення: **~15 хв**
- Коментар: відновлюємося досить швидко, але занадто часто доводиться це робити.
- Оцінка за DORA: **Medium–Low (не катастрофа, але є куди покращуватись)**

## 6. Який біль у нашій команді показали ці метрики?

- **Надто висока частка невдалих змін (CFR ≈ 96%)** — майже кожен деплой закінчується збоєм. Команда витрачає більше часу на виправлення помилок і перезапуски пайплайна, ніж на створення нової цінності.
- **Низька передбачуваність якості релізів** — дефекти виявляються вже у CI/CD, а не раніше (локальні тести, code review, staging). Кожен реліз сприймається як ризик, а не як «рутинний пуш у прод».
- **Часті мікро‑простої через відкат/відновлення (TTR ~15 хв)** — окремо взятий інцидент триває недовго, але їх багато; сумарно це «з’їдає» значну частину часу спринту.
- **Стрес та втрата довіри до пайплайна** — через часті падіння команда боїться деплоїтися наприкінці дня, відкладає релізи, робить хотфікси замість планової роботи.
- **Процес контролю якості слабший, ніж потрібно** — поточні практики тестування, code review та валідації змін не дають достатнього «захисного шару», тому біль зосереджений не у швидкості, а у якості.
